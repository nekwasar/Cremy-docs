## Issue 004: Referral Reward System

### Description
Implement the referral system where registered users earn credits by referring others. Handle both credit purchases and Pro subscriptions.

### Must Have
- Unique referral code per user
- Track referral source on signup
- On referred user credit purchase: 10% to referrer forever
- On referred user Pro subscription: 10 credits to referrer once
- Track referral history (who referred whom)
- Prevent self-referral
- Prevent duplicate referrals (same user referred multiple times)

### Logic
```
Referrer Reward Calculation:
- If referred user buys credits: add 10% of purchase amount to referrer
- If referred user subscribes Pro: add 10 credits to referrer (one-time)
- Reward continues for ALL future purchases (not just first)
```

### Dependencies
- Milestone: M1_2, M1_4
- Packages: Reference to CreditTransaction model

### Files
- src/lib/referral.ts
- src/app/api/referral/generate-code/route.ts
- src/app/api/referral/stats/route.ts
- src/models/Referral.ts

### Acceptance Criteria
- [ ] Each user has unique referral code
- [ ] 10% of credit purchases goes to referrer forever
- [ ] 10 credits given once for Pro referral
- [ ] Referral history tracked in database
- [ ] Self-referral prevented

## Issue 005: Pro Subscription Credit Allocation

### Description
Implement automatic credit allocation for Pro subscribers. Handle monthly and yearly plans with proper credit validity periods.

### Must Have
- On Pro monthly subscription: Add 200 credits to account
- On Pro yearly subscription: Add 2400 credits to account
- Track credit validity period separately from purchased credits
- Expire Pro credits at billing period end
- Different balance display: Total, Purchased (never expire), Pro (time-limited)
- Monthly cron job to expire old Pro credits
- Handle Pro renewal (add credits on renewal)

### Credit Validity Logic
```
User has:
- Purchased credits: NEVER expire
- Referral credits: NEVER expire  
- Reward credits: NEVER expire
- Pro credits: Valid only for billing period

On billing period end:
- Pro credits reset to 0
- If renewed: Add new Pro credits
```

### Dependencies
- Milestone: M1_4, M1_2 (subscription tracking)
- Packages: node-cron

### Files

### Acceptance Criteria
- [ ] Monthly: 200 credits added on subscription
- [ ] Yearly: 2400 credits added on subscription
- [ ] Pro credits expire at billing period end
- [ ] Renewals add new credits, reset validity
- [ ] Balance shows separate totals

## Issue 006: Rate Limiting by Credit Exhaustion

### Description
Implement rate limiting that triggers ONLY when users exhaust their credits. This is different from time-based rate limiting.

### Must Have
- Track credit usage per action
- When credits reach 0:
  - Complete CURRENT action
  - Block NEXT action
  - Show upgrade modal
- Per-action credit costs already defined in M1_4
- Graceful blocking (not abrupt disconnect)
- Show estimated credit cost BEFORE action
- Rate limit applies to both anonymous and registered users

### Rate Limit Trigger Logic
```
Action Requested
    ↓
Check credits >= cost
    ↓
Yes: Allow action, deduct credits
    ↓
No: Block action, show upgrade modal
```

### Dependencies
- Milestone: M1_4 (credit costs already defined)
- Packages: Already have rate limiting from M1_2

### Files
- src/lib/rate-limit.ts (update)
- src/middleware/check-credits.ts

### Acceptance Criteria
- [ ] Actions blocked when credits insufficient
- [ ] Current action completes even if credits reach 0 during it
- [ ] Upgrade modal shown on block
- [ ] Estimated cost shown before action

## Issue 007: Storage Toggle System

### Description
Implement the storage toggle system that allows users to choose between localStorage (browser) and MongoDB (cloud) storage. Handle data migration between storage types.

### Must Have
- User setting for storage preference (default: OFF for free users)
- Auto-enable MongoDB storage for Pro users
- For free users with >=10 credits: Option to enable storage
- Data migration: localStorage → MongoDB on toggle ON
- Data migration: MongoDB → localStorage on toggle OFF
- Storage used indicator in settings
- Deletion of old localStorage data after migration

### Storage Rules
```
Anonymous Users:
- Default: OFF (localStorage only)
- Stores: documents (encrypted preview only), credits, settings
- Auto-deletes after 24h if credits < 10
- Can turn ON: Saves to localStorage (still time-limited)

Registered Free Users:
- Default: OFF (can enable in settings)
- Option to enable: Saves to localStorage
- If >=10 credits: Data persists indefinitely in localStorage

Pro Users:
- Default: ON (MongoDB automatically)
- Can toggle OFF in settings (switches to localStorage)
```

### Dependencies
- Milestone: M1_3 (Document model has storage field)
- Packages: Already have localStorage

### Files
- src/lib/storage-toggle.ts
- src/app/api/user/settings/storage/route.ts
- src/services/data-migration.ts

### Acceptance Criteria
- [ ] Free users default: Storage OFF
- [ ] Pro users default: Storage ON
- [ ] Toggle ON migrates localStorage → MongoDB
- [ ] Toggle OFF migrates MongoDB → localStorage
- [ ] Data accessible after migration
